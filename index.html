Service crashes / restarts
A process dying is a kernel process event, not a filesystem event. inotify is blind to it. Covered by Layer 2 poller (≤5 min).

A Gemini key hitting its daily limit is a network / LiteLLM state change — no file is written. Covered by Layer 2 LiteLLM health poll.
✗
Memory count changes
New vectors added to Qdrant are stored in Qdrant's own data dir, not in ~/.openclaw/. Covered by Layer 2 Qdrant API poll.
✗
Network / port state changes
Services binding / unbinding ports, HTTP health endpoint going up or down — these are network events. Covered by Layer 2 service health probes.

Privacy Firewall
Every outbound message to drivers group or driver DMs runs sanitize_for_drivers() which strips:
  • customer_name, customer_phone, customer_whatsapp_id
  • customer_photo references
  • exact_address (street number) — only area/landmark allowed
  • payment_details, personal_notes, internal_comments
Any blocked field detected → send rejected, security event logged, human alerted immediately.

Mission Lifecycle — 5 Workflows
Workflow A — Customer → Mission
Trigger keywords in customers group:
transport deliver truck van sand iron goods cargo move ship load/unload hire
  1. Reply in group within 10s → "Check your DM ✋"
  2. Open private DM → send 7-question intake form
  3. Collect answers → validate → generate MIG-YYYYMMDD-NNN
  4. Create mission file in workspace/broker/missions/
  5. If dry_run: log + wait for APPROVE_MIG-XXX from endpoint
Workflow B — Broadcast to Drivers
Triggered by: broadcast_to_drivers(mission_id)
  1. Load mission → extract ALLOWED fields only
  2. Run sanitize_for_drivers() — raises PrivacyError on violation
  3. Format via driver_brief.md template
  4. Post to drivers group → offers remain OPEN until mission assigned or canceled
  5. If deadline approaching (15 min left) and <2 offers: post urgency reminder
Allowed fields: mission_id, product_type, weight, size, pickup_area, delivery_area, vehicle_required, deadline, distance_estimate_km
Workflow C — Driver Offer → Selection
Trigger: DM from driver containing mission ID
  1. Parse offer: price + vehicle + ETA
  2. Validate driver profile (create stub if new)
  3. Append to mission file under Offers section
  4. When ≥2 offers OR 15 min to deadline OR human signals ready → rank top 3
  5. DM endpoint: table of top 3 (price ↑, rating ↓, speed ↓)
  6. Wait for SELECT N reply — auto-select cheapest if timeout
Workflow D — Assignment & Notification
Trigger: SELECT N from endpoint
  1. Update mission file: status=assigned, driver, price, timestamp
  2. DM winning driver: pickup area + deadline + CONFIRMED request
  3. Notify customer: driver name, vehicle, ETA
  4. Archive rejected offers with reason not_selected
  5. Update MEMORY.md with outcome for future RAG
Customer contact is NEVER shared with driver directly — all coordination through broker.

REST POLICY
When a key hits its daily quota, it is placed on rest automatically.
LiteLLM cools it down for 24 h so it is never retried before reset.
At midnight Pacific Time (07:00 UTC), the LiteLLM proxy restarts and all daily quotas are released.
The pool rotates through remaining active keys with no user impact until all keys in a pool are exhausted.

quota-probe.py  MANUALv2
~/.openclaw/scripts/quota-probe.py
Multi-provider API rate-limit scanner. Directly probes Gemini (28 keys × 6 models) and Mistral (6 keys × 1 model), bypassing LiteLLM. Three test modes: Health (1 req/key — alive or dead?), Burst RPM (N simultaneous — find empirical ceiling), TPM Stress (16K-token payload — prove the 15K TPM wall). Saves results to ~/dex-dashboard/quota-scan.json after every run.
(no flags)Health scan all 28 Gemini + 6 Mistral keys across all models
--burst+ empirical RPM ceiling — fires burst_n simultaneous requests per model (model-aware: 38 for Gemma-3, 20 for 31B, 110 for embeddings, 10 for Mistral)
--tpm-stress+ TPM wall test — sends ~16K-token payload on one Gemma-3 key per model. Confirms 15K TPM/min limit. Costs ~16K tokens per tested model.
--fullRun all three tests (health + burst + tpm-stress) in one pass
--provider gemini|mistralProbe only one provider
--model MODEL_IDProbe only this model (e.g. gemma-3-27b-it, mistral-large-latest)
--pool POOL_NAMEProbe only this pool (e.g. gemma-chat, gemma-agentic, mistral-multimodal)
--key KEY_NAMEProbe only one key (e.g. GEMINI_API_KEY_5) — useful to debug a specific key
--samples NBurst-test N keys per model instead of default 3 — use --samples 28 for full pool burst
--jsonMachine-readable JSON output only — pipe to jq or redirect to file
