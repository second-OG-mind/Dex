HYBRID ARCHITECTURE
LAYER 1
👁
inotify
DAEMON
Kernel filesystem events — fires in <2 seconds
The Linux kernel notifies dex-watcher.service the instant any file inside ~/.openclaw/ is written or created. A 60-second debounce prevents spam from rapid successive writes (e.g. the gateway's own file watcher writing telemetry on startup).
dex-watcher.service Restart=always debounce 60s
<2s
LAYER 2
⏱
SMART
POLLER
Diff-on-change cron — fires every 5 minutes
watch-changes.sh snapshots service state, LiteLLM healthy/unhealthy endpoint counts, and Qdrant memory point counts. It compares the snapshot to the last known state — only pushes if something changed. Covers what inotify cannot: process crashes and API quota exhaustion.
*/5 cron diff-on-change watch-changes.sh
≤5m
LAYER 3
💓
HEARTBEAT
CRON
Always-on full update — fires every 15 minutes
update-status.sh runs unconditionally every 15 minutes, collects the full system snapshot, writes data.json, and pushes to GitHub Pages. This is the catch-all floor — anything missed by the layers above is guaranteed to appear within 15 minutes.
*/15 cron always fires update-status.sh
≤15m
inotify CAN DETECT
✓
API key rotation
Any write to ~/.openclaw/.env — new keys added, existing keys changed or rotated — fires the watcher instantly.
✓
Config changes
Writes to openclaw.json — model routing, dmPolicy, channel config, plugin enable/disable — detected in <2s.
✓
Plugin config changes
Any *.plugin.json write under ~/.openclaw/extensions/ — plugin version bumps, config edits, new plugin installs.
✓
Knowledge base changes
Any *.md file written in the openclaw tree — Cognee indexed docs, agent memory exports added as files.
inotify CANNOT DETECT
✗
Service crashes / restarts
A process dying is a kernel process event, not a filesystem event. inotify is blind to it. Covered by Layer 2 poller (≤5 min).
✗
API key quota exhaustion
A Gemini key hitting its daily limit is a network / LiteLLM state change — no file is written. Covered by Layer 2 LiteLLM health poll.
✗
Memory count changes
New vectors added to Qdrant are stored in Qdrant's own data dir, not in ~/.openclaw/. Covered by Layer 2 Qdrant API poll.
✗
Network / port state changes
Services binding / unbinding ports, HTTP health endpoint going up or down — these are network events. Covered by Layer 2 service health probes.

REST POLICY
When a key hits its daily quota, it is placed on rest automatically.
LiteLLM cools it down for 24 h so it is never retried before reset.
At midnight Pacific Time (07:00 UTC), the LiteLLM proxy restarts and all daily quotas are released.
The pool rotates through remaining active keys with no user impact until all keys in a pool are exhausted.
ROUTING RULES
Text requests → input-router → openai plugin → LiteLLM → Gemini Flash pool (12 keys)
Media / Files / Voice → input-router → mistral plugin → Mistral Large pool (5 keys)
Memory embeddings → LiteLLM → Gemini Embedding pool (12 keys)
Mistral is never used as fallback for text. If all Gemini keys are resting, the gateway waits for midnight reset.
